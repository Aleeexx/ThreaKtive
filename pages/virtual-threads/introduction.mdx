# Virtual Threads

## Project Loom 

Die Einführung von Virtual Threads ist Teil vom "Project Loom" und wurde offiziell mit Java 21 herausgebracht und eingeführt. [^1]
Es ist eine sehr große und bedeutsame Erneuerung. Virtual Threads bieten eine einfachere und effizientere Möglichkeit,
mit nebenläufigen Programmierparadigmen umzugehen. Bei dem Model „Request-per-Threads“ kommt es bei vielen Anfragen mit
blockierenden Aufrufen zu einem Platform Thread Problem, diese sind schwergewichtet und es können nicht beliebig viele erzeugt werden.
Diese Problematik konnte in der Vergangenheit mit der reaktiven Programmierung mithilfe von Frameworks gelöst werden.
Da bei diesem Programmierparadigma nicht blockierend gewartet wird. Auf dieser Website sind unsere Erfahrungsberichte
mit einer Java Backend Umstellung auf eine reaktive Version. Kurz zusammengefasst kann gesagt werden, es ist ein sehr
großer Aufwand und es müssen die Entwickler mit passendem Knowhow vorhanden sein. Zudem werden die Lesbarkeit und das
Debugging deutlich schwieriger und komplexer. Die genauen Informationen von dem Erfahrungsbericht können in [ThreaKtive - Code Beispiele - Erfahrungsbericht](/reactive-programming/experience-report) eingesehen werden.
Dies könnte der Grund sein, weshalb viele Java Backend Anwendungen nicht auf eine reaktive Version mithilfe von Frameworks umgestellt wurden.
Mit der Einführung von virtuellen Threads wurde eine nicht blockierende Anwendung zu integrieren deutlich einfacher. Es können deutlich mehr virtuelle Threads als Platform Threads erzeugt und verwaltet werden. Ein wichtiger Ansatz der virtuellen Threads ist das spezielle Objekt „Continuation“, dieses Schlüsselkonzept wird auf den weiteren Unterseiten näher betrachtet. [^2] [^3]
